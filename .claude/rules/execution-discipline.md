# Дисциплина выполнения

Эти правила устраняют системные паттерны фрикции, выявленные в /insights.

## Выполнение команд

- Выполняй команды напрямую — не предлагай запустить самостоятельно
- Исключение: деструктивные операции (rm -rf, DROP TABLE, reset --hard) — спросить подтверждение
- Если команда незнакома → проверить синтаксис через --help перед запуском, не угадывать флаги

## Проверка до действия

- Перед любым действием прочитать релевантные файлы — не угадывать структуру
- Перед генерацией артефакта (Excel, Marp, отчёт) — написать checklist всех метрик/слайдов/строк и получить подтверждение
- "Готово" = каждый пункт задачи явно проверен и подтверждён, не просто написан код

## Pre-flight для внешних интеграций

Перед интеграцией с любым внешним сервисом (MCP, API, OAuth, БД):
1. Проверить plan/tier limits — не начинать если лимит заблокирует use case
2. Перечислить auth requirements и нужен ли port-forwarding
3. Найти known version incompatibilities со стеком
4. Перечислить silent failure modes которые нужно проверить после

## Scope перед началом

- При неясном запросе — уточнить "do X, not Y" границу до начала работы, не после первой попытки
- Не добавлять своих design ideas если не просили — следовать spec буквально
