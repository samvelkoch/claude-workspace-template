## Baseline-оракул

Baseline — эталонная точка отсчёта. Все эксперименты меряются относительно него.

- **Baseline модель**: [например, LightGBM дефолтные параметры, feature set v0]
- **Baseline метрики**: [например, MRR=0.31, hit@3=0.45 на hold-out]
- **Baseline скрипт**: `src/baseline.py`

**Правила:**
- Никогда не коммитить изменение роняющее метрику ниже baseline без понимания причины.
- Перед экспериментом записать ожидаемое направление изменения.
- Улучшение baseline — только после валидации на полных данных.
- Если baseline не воспроизводится → остановиться, разобраться, ничего не делать дальше.
- Никогда не подгонять пороги чтобы метрика выглядела лучше без понимания причины.
  Если что-то выглядит неправильно — трейсить назад до данных, не патчить вывод.

## Режимы валидации

| Режим | Данные | Когда |
|-------|--------|-------|
| **quick** | `data/sample/` или 10% стратифицированно | После каждого изменения |
| **full** | Полный датасет | Перед коммитом результатов |

- Fast: детерминированный (фиксированный seed, одни и те же строки)
- Fast: покрывать крайние случаи — топ/хвост ранжирования, редкие классы, все сегменты

## Гигиена вывода

У LLM конечное контекстное окно. Шумный вывод вытесняет полезную информацию.

- Никогда не дампить DataFrame в stdout. Печатать: `shape=(1M, 47), nulls=0, dtypes ok`
- Никогда не печатать полные массивы или длинные value_counts. Агрегаты: top-3, перцентили, counts.
- Verbose вывод → `logs/`, не в stdout.
- После эксперимента: одна строка с датой, метриками, дельтой к baseline.

Хорошо: `[2026-03-25] price_relative: MRR=0.334 (+0.024 к baseline) | quick`
Плохо: дамп DataFrame на тысячи строк в чат

## Параллелизм

**Легко параллелизируется:**
- Разные группы фичей (временные, ценовые, маршрутные, история)
- Разные семейства моделей (LightGBM vs CatBoost vs XGBoost)
- Разные сегменты данных

**Блокирует всё (нельзя параллелить пока не решено):**
- Утечка данных — все результаты под вопросом до разрешения
- Изменения схемы данных
- Изменение определения целевой переменной

**Claiming задач** — при старте параллельной работы записать в "В работе":
```
- [2026-03-25 14:30] @agent-features: временные фичи — lag_7, lag_14, rolling_mean
- [2026-03-25 14:30] @agent-model: CatBoost baseline — дефолт, тот же feature set
```
Перед стартом проверить PROGRESS.md — не дублировать.

## Специализированные роли агентов

Для сложных задач запускать специализированных агентов:

- **feature-engineer** — исследует и реализует одну группу фичей
- **model-tuner** — гиперпараметры на фиксированном feature set
- **data-validator** — проверка утечек, схемы, сдвигов распределений
- **results-documenter** — обновляет PROGRESS.md, пишет summary эксперимента
- **code-reviewer** — проверяет утечки, векторизацию, group-aware сплиты

Назначать разные файлы/модули разным агентам чтобы не конфликтовать.

## Дисциплина коммитов

- Один коммит = одна логическая единица: группа фичей, эксперимент, баг-фикс
- Каждый коммит: прошёл baseline check + результаты записаны в PROGRESS.md
- Никогда не коммитить: raw данные, credentials, ноутбуки с полным выводом, >10MB
- Формат: `[experiment|feature|fix|refactor] краткое описание`

## Правила
- Только векторизованные операции — никаких циклов по большим данным
- Сначала quick mode, full только перед коммитом результатов
- После каждой значимой единицы работы: обновить PROGRESS.md и сделать коммит
- Если подход не сработал: записать в Dead ends с причиной до перехода к следующему
- Валидировать данные на каждом этапе пайплайна, не только финальный результат
- Сравнивать минимум 2 архитектуры перед выводами
