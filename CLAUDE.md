## Baseline-оракул

Baseline — эталонная точка отсчёта. Все эксперименты меряются относительно него.

- **Baseline модель**: [например, LightGBM дефолтные параметры, feature set v0]
- **Baseline метрики**: [например, MRR=0.31, hit@3=0.45 на hold-out]
- **Baseline скрипт**: `src/baseline.py`

**Правила:**
- Никогда не коммитить изменение, которое роняет метрику ниже baseline, без понимания причины.
- Перед началом эксперимента записать ожидаемое направление изменения.
- Улучшение baseline — только после валидации на полных данных.
- Если baseline не воспроизводится → остановиться, разобраться, ничего не делать дальше.
- Никогда не подгонять пороги чтобы метрика выглядела лучше без понимания причины.
  Если что-то выглядит неправильно — трейсить назад до данных, не патчить вывод.

## Режимы валидации

Два режима — fast при разработке, full перед коммитом результатов:

| Режим | Данные | Когда использовать |
|-------|--------|--------------------|
| **fast** | `data/sample/` или 10% стратифицированно | После каждого изменения |
| **full** | Полный датасет | Перед коммитом результатов |

- Fast режим должен быть детерминированным (фиксированный seed, одни и те же строки)
- Fast режим должен покрывать крайние случаи: топ/хвост ранжирования, редкие классы, все сегменты

## Гигиена вывода (экономия контекстного окна)

У LLM конечное контекстное окно. Шумный вывод вытесняет полезную информацию.

**Правила:**
- Никогда не дампить DataFrame в stdout. Печатать: `shape=(1M, 47), nulls=0, dtypes ok`
- Никогда не печатать полные массивы или длинные value_counts. Агрегаты: top-3, перцентили, counts.
- Verbose вывод логировать в `logs/`, не в stdout.
- После каждого эксперимента: одна строка с датой, метриками, что изменилось.
- При сравнении моделей: показывать дельту к baseline, не только абсолютные числа.

Хорошо:
```
[2026-03-25] feature_price_v2: MRR=0.334 (+0.024 к baseline), hit@3=0.471 (+0.021) | fast mode
```
Плохо:
```
              feature1  feature2  target
0             0.234     1.123     1
[1000000 rows × 47 columns]
```

## Параллелизм

Часть работы независима и может быть распределена между агентами:

**Легко параллелизируется (независимо):**
- Разные группы фичей (временные, ценовые, маршрутные, история пользователя)
- Разные семейства моделей (LightGBM vs CatBoost vs XGBoost)
- Разные сегменты (корпоративные vs leisure, внутренние vs международные)

**Блокирует всё остальное (нельзя параллелить пока не решено):**
- Утечка данных (data leakage) — все результаты под вопросом до разрешения
- Изменения схемы данных — ломает все downstream фичи
- Изменение определения целевой переменной

**Claiming задач:** при запуске параллельной работы добавить в "В работе":
```
- [2026-03-25 14:30] @agent-features: временные фичи — lag_7, lag_14, rolling_mean
- [2026-03-25 14:30] @agent-model: CatBoost baseline — дефолтные параметры, тот же feature set
```
Перед стартом проверить PROGRESS.md чтобы не дублировать.

## Дисциплина коммитов

- Один коммит = одна логическая единица: одна группа фичей, один эксперимент, один баг-фикс
- Каждый коммит должен: пройти baseline check + иметь результаты в PROGRESS.md
- Никогда не коммитить: raw данные, credentials, ноутбуки с полным выводом, файлы >10MB
- Формат сообщения: `[experiment|feature|fix|refactor] краткое описание`

## Специализированные роли агентов

Для сложных экспериментов запускать специализированных агентов:

- **feature-engineer**: исследует и реализует одну группу фичей
- **model-tuner**: поиск гиперпараметров на фиксированном feature set
- **data-validator**: проверка утечек, проблем со схемой, сдвигов распределений
- **results-documenter**: обновляет PROGRESS.md, пишет summary эксперимента
- **code-reviewer**: проверяет утечки, векторизацию, group-aware сплиты

Назначать разные файлы/модули разным агентам чтобы не конфликтовать.

## Правила
- Только векторизованные операции — никаких циклов по большим данным
- Group-aware сплиты: GroupKFold по ranker_id (никогда не мешать группы)
- Сначала fast mode, полные данные только перед коммитом результатов
- После каждой значимой единицы работы: обновить PROGRESS.md и сделать коммит
- Если подход не сработал: записать в Dead ends с причиной до перехода к следующему
- Валидировать данные на каждом этапе пайплайна, не только финальный результат модели
- Сравнивать минимум 2 архитектуры модели перед выводами
